# 🏃‍♂️ Project 330: AI Running Coach & Dashboard Specification

본 문서는 Strava API와 Gemini API를 활용하여 사용자의 러닝 데이터를 실시간으로 동기화하고, AI 모델을 통해 개인 맞춤형 주간 계획(Week Plan) 및 다음 훈련 추천(Next Running)을 유기적으로 변동시키는 **AI 러닝 코치 대시보드 시스템**의 명세서 및 아키텍처 정리 파일입니다.

---

## 1. 프로젝트 개요 (Project Overview)
* **명칭:** Project 330 Dashboard
* **목적:** 가상 서버 인프라(GitHub Actions)와 러닝 플랫폼 API(Strava), 생성형 AI(Gemini API)를 결합하여 개발자나 러너가 모바일 앱처럼 언제 어디서나 확인할 수 있는 **독립형 주황색 다크 테마 대시보드 웹사이트**를 제공합니다.
* **핵심 가치:** 정적인 고정 텍스트 반복에서 탈피하여, 실제 러닝 실적과 컨디션 데이터에 기반해 주간 목표 마일리지를 지능적으로 분배하고 실시간으로 가이드를 수정하는 Dynamic AI 코칭을 실현합니다.

---

## 2. 지금까지 구현된 기능 (Current Implementation)

현재까지 안정적으로 빌드되어 깃허브 가상 머신 및 웹 서버 상에서 동작하는 인프라 스펙은 다음과 같습니다.

### 2.1. 독립형 웹 호스팅 및 배포 자동화
* **GitHub Pages 개통:** 리포지토리의 `index.html` 파일을 기반으로 외부에서 접속 가능한 단독 웹사이트 주소 개통 완료.
* **CI/CD 파이프라인 구축 (`coach_sync.yml`):**
  * 10분 주기(`*/10 * * * *`)로 Strava 백엔드와 자동 동기화.
  * 매일 밤 자정 및 새벽 시간대(한국 시간 기준 00시, 01시, 02시)에 자동으로 깨어나 대시보드를 리셋하고 새로운 날짜의 루틴을 반영하는 크론탭(Crontab) 스케줄러 내장.
* **무결점 Python 기반 파일 생성:** 인코딩 오류 및 마크다운 치환 에러를 방지하기 위해 `coach.py` 내부에서 직접 프리미엄 다크 테마 HTML 코드를 렌더링하도록 일체화.

### 2.2. 프론트엔드 UI/UX 디자인 사양
* **Premium Dark Theme:** 시인성이 높고 눈 피로가 적은 `#121212`(배경) 및 `#1e1e1e`(컨테이너) 다크 모드 적용.
* **Brand Accent Color:** 스트라바 고유의 정체성을 상징하는 주황색 시그니처 컬러(`#FC4C02`)를 헤드라인, 인라인 강조, 테두리에 배치.
* **데이터 시각화:** QuickChart API를 연동하여 월간/주간 마일리지를 나타내는 SVG/인라인 막대 차트 그래픽 구현.
* **모바일 최적화:** 뷰포트(`viewport`) 반응형 메타 태그를 적용하여 아이폰 및 갤럭시 등 스마트폰 브라우저 세로 화면에 완벽하게 핏(Fit)되도록 컨테이너 최대 너비를 `500px`로 제한.

### 2.3. 한국 표준시(KST) 및 타임존 고정
* 깃허브 가상 서버가 위치한 해외 리눅스 환경(UTC)의 영향을 받지 않고, 파이썬 내에서 `timezone(timedelta(hours=9))` 연산을 수행하여 **대한민국 시간** 기준으로 `Last Sync` 및 `Daily Running` 날짜가 칼같이 매칭되도록 설계 완료.

## 3. 고도화 및 추가 구현 기능 (Upgraded AI Features)

기존의 단순 정적 텍스트 복사기 구조를 타파하고, 데이터 지능형 동적 대시보드로 전환하기 위한 상세 사양입니다.

### 3.1. [신설] 실시간 주간 러닝 계획 구역 (Week Plan Section)
* **위치:** 대시보드 내 '위클리 마일리지(Weekly Mileage)'와 '데일리 러닝(Daily Running)' 섹션 사이.
* **내용:** 월요일부터 일요일까지의 일별 조깅 마일리지 분배 표 및 훈련 종류(예: 조깅, 빌드업, 인터벌, 휴식) 명시.
* **연산 및 분배 모델:** **Gemini 1.5 Pro / Flash Model**
  * **분배 로직 (AI의 추론 방식):** 가상 서버가 트리거될 때 아래의 변수들을 Gemini API에 프롬프트 데이터로 주입합니다.
    $$	ext{Variables} = \{ 	ext{주간 목표 마일리지 (예: 50km)}, 	ext{현재 요일}, 	ext{이번 주 누적 소화 마일리지}, 	ext{최근 러닝 페이스 및 강도} \}$$
    Gemini 모델은 이 데이터를 기반으로 오버트레이닝 여부를 연산한 뒤, 남은 요일 동안 가장 안전하고 효율적으로 목표 마일리지를 달성할 수 있는 **일별 유동적 마일리지 분배표**를 실시간으로 예측·계산하여 출력합니다.

### 3.2. 지능형 넥스트 러닝(Next Running) 추천 엔진
* **개념:** 고정된 정적 문구가 아닌, 오늘의 운동 실적과 신체 피드백을 기반으로 동작하는 개인형 AI 코칭.
* **작동 매커니즘:**
  * 만약 전날 달린 페이스가 평소보다 오버페이스였거나 심박수가 과도하게 높았다면, 다음 날 Next Running 구역은 자동으로 **"근육 데미지 누적 감지: 5km 미만의 가벼운 리커버리 조깅 또는 완전 휴식 권장"**으로 변경됩니다.
  * 반대로 목표 마일리지 달성률이 저조하고 컨디션이 양호하다면 강도를 높인 **"7km 빌드업 러닝 또는 지속주 훈련 추천"**으로 문구와 타겟 심박수를 실시간 연산하여 교체합니다.

### 3.3. 실시간 양방향 연동 체제 (Dynamic Feedback Loop)
* **작동 구조:**
  1. **실적 분석:** Strava API를 통해 들어온 러닝 기록 데이터 파싱.
  2. **컨디션 주입:** 사용자가 맥북 터미널이나 파일 수정을 통해 당일의 몸 상태(예: 무릎 통증 있음, 컨디션 좋음 등)를 `condition.txt` 같은 경량 로그 파일에 남기면 파이썬이 이를 함께 취합.
  3. **AI 통합 추론:** 취합된 실적 데이터와 신체 상태 데이터를 Gemini API에 동시 전달.
  4. **주간 계획 및 추천 실시간 동적 변동:** AI가 도출한 데이터에 맞춰 **Week Plan 표와 Next Running 추천 문구가 실시간으로 유기적으로 갱신**되며 대시보드(index.html)가 다시 구워짐.

## 4. 데이터 흐름 및 아키텍처 (Data Flow Diagram)

```
[Strava API] --------(최근 러닝 데이터 추출)------┐
                                                 v
[사용자 피드백] ------(컨디션/통증 여부 입력)----> [coach.py (GitHub Actions)]
                                                 │
                                           (AI 통합 프롬프트 전송)
                                                 │
                                                 v
                                          [Gemini API 연산]
                                    (Week Plan 분배 및 Next 추천 생성)
                                                 │
                                                 v
[개통된 대시보드 주소] <-- (Pages 배포) -- [Premium index.html]
```
